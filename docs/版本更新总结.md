## 自动化测试（pytest）

项目引入基于 pytest 的自动化测试体系，包含 **186 个测试用例**，全部 **2.9 秒内完成**。

**测试架构**：四层架构，damage_calc.py 是唯一真相来源

```
damage_calc.py  ← 唯一真相来源（纯函数，零 GUI 依赖）
    │               所有公式、常量、筛选逻辑都在这里
    │
    ├── import ── WWDmgCalc.py          主程序 GUI
    │              _matches_filter() → damage_calc.matches_filter()
    │              EnemyDefensePage.recalc() → damage_calc.calc_defense_zone()
    │              EnemyResistancePage._recalc() → damage_calc.calc_resistance_zone()
    │
    └── import ── tests/test_damage_formula.py  测试
                  from damage_calc import calc_defense_zone, ...
                  assert calc_defense_zone(90, 100, 0) == ...
```

## Git 版本管理

项目使用 Git 管理版本。关键操作：

```bash
# 日常存档（每次改完代码做一次）
git add .
git commit -m "改动说明"

# 出问题回退
git log --oneline          # 查看历史
```

  **现象**：启用「基础数值调整」后，计算结果页的计算过程框顶部出现一条 QLabel 提示文字「▸ 已启用手动填写基础数值」，本意是提醒用户目前在用手动值。但 QLabel 是放在 _process_layout 最顶部的独立 Widget，导致 HTML 表格还没渲染时它就卡在第一行位置，形成「置顶」效果，视觉上割裂了计算过程表格。同时结果详细窗口里**完全没有**这条提示——因为 ResultDetailDialog 读的是卡片缓存的 `process_html`，而该缓存由 `_build_card_process_html` 生成，调用 `_render_process_html` 时漏传了 `base_override_active` 参数。

  **根因链**：
  1. 提示被分到三个独立的渲染路径，各自缺失或重复，从未对齐
  2. `ResultPage.compute()` → `_render_process_html(base_override_active=True)` 在 HTML 的 `<br>` 中内嵌提示 ✅，但同时又有一个 QLabel Widget 被加在 layout 顶部 ❌ 形成双份+置顶
  3. `ResultDetailDialog._rebuild_process_html()` 自己重新构建 HTML，读 `self._page._base_override_enabled` 埋 `<tr>` 行在基础数值行下面 ✅，但这只在缓存为空时执行
  4. `ResultListPage._build_card_process_html()` → `_render_process_html(is_light=False)` **未传** `base_override_active` ❌，导致缓存缺失提示

  **修复**：
  1. 移除 ResultPage 中的 QLabel Widget（`_base_override_hint`），提示完全改为 HTML 内嵌
  2. `_build_card_process_html` 补传 `base_override_active` + `computed_base_zone`，缓存中自带提示
  3. `_rebuild_process_html` 将 `<p>` 标签改为 `<tr>` 标签，让提示作为表格行紧接在「基础数值」行下面

## v1.0 数据流精度与一致性修复（2026-06-15）

### float 精度清洗（_round_floats）
- **fix**：保存/导出时递归清洗所有 float 到 4 位小数 — 消除 IEEE 754 垃圾尾数（如 536.799...→536.79→修正为 536.8、489.44000000000005→489.44）
- **原理**：QDoubleSpinBox decimals=4 → alue() 返回 IEEE 754 双精度原始值。json.dumps() 直接输出 15 位精度，用户输入的 536.8 在 JSON 中显示为 536.79。_round_floats() 在落盘前把 float 统一 round(4)，恢复用户看到的数字。计算不改变，只在序列化时清洗。
- **影响**：预设 JSON 和存档文件中的数值恢复为真实有效值，不会丢失精度——超出 4 位的尾数全是 IEEE 754 噪声，没有有效信息。
